After /survey catalogued 22 smells, the next question wasn't how to fix them — it was in what order. In a legacy monolith, everything touches everything. You don't pick randomly. You map.

The situation: 22 open smells, no order

14 critical (🔴), 5 medium (🟠), 3 low (🟡). SQL injection in 4 different route groups. MD5 passwords. Admin endpoints with zero auth. A God module at 368 lines.

Fixing them in random order is how you introduce regressions. The map phase exists to prevent exactly that.

The task: let Claude draw the dependency graph

/map reads the full codebase and the smell catalog, then produces a structured risk map: red zones (touch last, highest coupling), orange zones (medium risk, touch in order), green zones (safe to isolate and fix first).

The output goes to logs/audit/map.md — versioned, not ephemeral. Claude doesn't decide this in a chat. It decides it in a document that survives the session.

The result: refactor order decided

Auth first — it's where SQL injection and MD5 passwords live. It's also the highest blast radius if something breaks. Fixing auth properly (SQLAlchemy + bcrypt + parameterized queries) unlocks every other route that depends on identity.

Gate validated. Phase committed to git. The state machine moved from map: not_started to map: completed.

The lesson: order is architecture

The map phase isn't about understanding the code — /survey did that. It's about understanding the coupling. Which smell, when fixed, unblocks the most other smells? That's the refactor order. And it's not intuitive — it has to be derived.
